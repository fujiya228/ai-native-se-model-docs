## なぜ分解原則が必要か

[9段階のライフサイクル](/lifecycle/nine-stages/)（L1）を詳細化する際、L2を「できるだけ細かく」するのは誤りである。L2を均等に細かくすると、以下の問題が起きる。

- ステップ数が増えすぎて全体像が見えなくなる
- 実際には違いがないステップを無理に分けることになる
- RACI、裁量レベル、依存関係を載せたときに差が出ない
- 表が肥大化し、更新しづらくなる

L2の正しい粒度は「できるだけ細かく」ではなく、**違いが生じる最小粒度**である。

## 分割原則

L2は、以下のいずれかが変わるなら分ける。

### 1. 実行主体が変わるなら分ける

同じL1の中でも、仕事を前に進める主体が切り替わるなら分ける価値がある。

**例**: Verification & Review の中で、自動検証（Automation）、人レビュー（Human）、差し戻し判断（Human）は実行主体が異なるため、L2として分ける。

### 2. 出力物が変わるなら分ける

結果として残すアーティファクトが変わるなら、後続の受け渡しや依存設計が変わるため分ける。

**例**: Specification の中で、「論点整理」と「受け入れ条件の明文化」は後続へ渡すものが異なる。

### 3. 次に進む条件が変わるなら分ける

品質ゲートや承認条件が変わるなら分ける。

**例**: Implementation の中で、「変更実装」と「ローカル整合確認」は連続して見えるが、後者には「次工程へ出せる最低品質」というゲートが発生する。

### 4. 依存先/依存元が変わるなら分ける

先行条件や同期条件が変わるなら分ける。

**例**: Decomposition の中で、「変更単位の分解」と「実行順序の整理」は扱う依存関係の種類が異なる。

### 5. 成否を見る指標が変わるなら分ける

何を測るべきかが変わるなら分ける。

**例**: Release & Operate の中で、「デプロイ」ではリードタイムや反映成功率が重要で、「監視・初動復旧」では検知時間と復旧時間が重要になる。

## 停止原則

以下の**すべてが同じ**なら、それ以上分けない。

- 実行主体が同じ
- 入出力アーティファクトが同じ系統
- 制御点が同じ
- 依存関係が同じ
- 測定観点が同じ

<Aside type="tip">
L2を細かくしても独立した設計判断が増えないなら、その分割は説明上の細分化にとどまり、モデルとしての価値は増えない。
</Aside>

## 簡易チェック

新しいL2を追加したくなったときは、以下を確認する。

| チェック項目 | 変わるなら分ける |
|---|---|
| この分解で実行主体は変わるか？ | Yes → 分ける |
| この分解で出力物は変わるか？ | Yes → 分ける |
| この分解で次に進む条件は変わるか？ | Yes → 分ける |
| この分解で依存先/依存元は変わるか？ | Yes → 分ける |
| この分解で成否を見る指標は変わるか？ | Yes → 分ける |

1つでも「変わる」があるなら、L2として独立させる価値がある。すべて「変わらない」なら、説明文や補足例として扱う。

## 実務上の運用ルール

| ルール | 内容 |
|---|---|
| **L1は必須、L2は任意** | L1は全体の背骨として固定する。L2は必要な箇所だけ定義する |
| **L2の数を揃えない** | 各L1に機械的に同数のL2を置かない。変化点が多いL1は深くし、差が少ないL1は粗く保つ |
| **依存関係はL2表に押し込まない** | 依存関係は「属性」ではなく「関係」。別表（[依存関係ビュー](/views/view-4-dependency/)）で管理する |
| **図は説明用、表は正本** | Mermaid等の図は理解を助けるが複雑化しやすい。正本はL2定義表と依存関係表に置く |
| **L2は基準形であり、現場で増減させる** | 参照モデルの基準形を起点に、チーム実務に合わせて統合・追加する |

<Aside>
本モデルのL2定義（[L2分解と設計属性](/lifecycle/l2-decomposition/)）は「完備一覧」ではなく「基準形の例」である。この原則に基づいて、自分のコンテキストに合わせてテーラリングすることを前提としている。
</Aside>
