## なぜモデルの性格を明示するのか

モデルを使う人は、暗黙のうちに「これは現状を記述しているのか、あるべき姿を規定しているのか、変革の方向を示しているのか」を判断しなければならない。この区別が曖昧だと、記述を規範と読み違えたり、変換の指針を絶対的な正解と誤解したりする。

本モデルは**テーラリング可能な参照モデル**である。特定の組織や技術スタックの「正解の工程表」ではない。仕事の流れを言語化し、設計判断を明示し、異なるコンテキストに適応させるための共通フレームである。

## 性質

本モデルには、以下の性質が同居している。

| 性質 | 英名 | 答える問い | 提供するもの |
|---|---|---|---|
| 記述的 | Descriptive | 仕事はどう流れているか？ | 語彙と枠組み |
| 規範的 | Normative | 仕事を設計するとき何を定義すべきか？ | 構造と原則 |
| 変換的 | Transformational | AI 導入で何がどう変わるか？ | 変換の視点 |

### 記述的（Descriptive）

仕事がどう流れているかを構造的に捉えるための**語彙と枠組み**を提供する。

- ライフサイクル
- 基底モデルの要素
- ビュー

これらは現実の仕事の流れを分析・記述するために使える道具である。「こうすべき」ではなく「こう見える」を語るための言葉を与える。

### 規範的（Normative）

仕事を設計する際に「**何を定義すべきか**」を示す。

- 各ステップに対してステップ構成を定義すべきという構造自体が規範である
- L2 の分割原則（差が生じるところだけ詳細化する）も規範である

ただし、各属性の**中身**（誰を Responsible にするか、どの裁量レベルにするか等）は規定しない。枠は固定するが、値は固定しない。

### 変換的（Transformational）

AI エージェント導入による仕事の再設計を扱うための**視点**を内蔵している。

- 実行主体と責任主体の分離
- 裁量レベル（L0〜L4）
- As-Is / To-Be / Validation ビュー

これらは AI 導入前後の差分を設計するために存在する。「AI を入れると何が変わるか」「どこまで委譲し、どこにガードレールを置くか」を考えるための道具である。

## 性質の関係

これらの性質は矛盾するものではなく、**層をなしている**。

```
記述的（語彙と構造）     ← 土台
  ↑
規範的（何を定義すべきか） ← 記述的の上に載る
  ↑
変換的（AI導入でどう再設計するか） ← 規範的の上に載る
```

記述的な枠組みがなければ規範を語れない。規範がなければ変換の設計ができない。本モデルは記述の語彙を土台に、設計の規範を載せ、その上に AI ネイティブ化という特定の変換を扱う構造になっている。

<Aside type="tip">
モデルを読むとき、各ページがどの性質を主に持つかを意識すると理解しやすい。たとえば、[ライフサイクル](/lifecycle/nine-stages/)は規範的、[自己批評と未解決論点](/appendix/critique-and-gaps/)は記述的な性質が強い。
</Aside>

## モデルが固定するものと固定しないもの

### 固定するもの（テーラリングの枠組み）

| 要素 | 性質 |
|---|---|
| ライフサイクルの枠組み（L0 → L1 → L2） | 記述的 |
| 基底モデルの要素 | 記述的 |
| ステップ構成の枠 | 規範的 |
| ビューの体系 | 規範的 |
| L2 の分割原則・停止原則 | 規範的 |
| 実行主体と責任主体の分離 | 規範的 + 変換的 |

これらはどのコンテキストでも共通の語彙・構造として機能する。テーラリングの対象ではなく、テーラリングの枠組みそのものである。

### 固定しないもの（テーラリング対象）

| 要素 | 判断者 |
|---|---|
| 実行主体の割当（AI か Human か） | チーム / 組織 |
| 裁量レベルの選択（L0〜L4） | チーム / 組織 |
| RACI の中身 | チーム / 組織 |
| ゲート条件の厳しさ | チーム / 組織 |
| 測定指標の具体値 | チーム / 組織 |
| 使用ツール・基盤 | チーム / 組織 |

これらはコンテキストに応じて埋める調整項目である。モデルは項目の枠を定義するが、値は規定しない。

## テーラリングの最小セット

全ビューを一度に埋める必要はない。まず以下を押さえれば「誰が何をどの順序で進め、どこで待ちが生じるか」が見える。

- ライフサイクルビューで対象段階の開始・終了条件を定義する
- 実行設計ビューでRACIと裁量レベルを埋める
- 依存関係ビューで主要な依存を特定する

## 本モデルがカバーしないこと

誤解を防ぐために、本モデルの対象外を明示する。

- **特定の組織の工程表ではない** --- L1 は参照形であり、そのまま採用することを前提としていない
- **AI の導入ガイドではない** --- 「AI を入れるべきか」「どのツールが良いか」は答えない。AI を入れると決めた後の設計の枠組みを提供する
- **成熟度モデルではない** --- レベル1〜5のような段階評価は行わない
- **プロセス改善フレームワークではない** --- CMMI や SPICE のような組織プロセスの評価・改善体系ではない
- **完成品ではない** --- 適用しながら育てる前提の参照モデルであり、フィードバックによって進化する
