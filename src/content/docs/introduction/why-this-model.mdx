## AIエージェント時代の問い

AIエージェントがコードを書き、テストを実行し、PRを作成する。この変化はすでに起きている。しかし、多くのチームが直面しているのは「どのツールを使うか」ではなく、もっと根本的な問いである。

- 仕事の流れのどこにAIを入れるのか
- どこまでの裁量を委ね、どこに人の判断を残すのか
- 安全に回すために何を設計しておくべきか
- AI導入前後で何が変わり、何が変わらないのか

これらの問いに答えるには、「ツールの機能比較」でも「AIに全部任せる/任せない」の二択でもなく、**ワークフロー全体を構造的に捉え、設計判断を明示するための枠組み**が必要になる。

## 既存のフレームワークでは足りない

ソフトウェアエンジニアリングには、仕事の流れを捉えるためのフレームワークがすでに多数存在する。

| フレームワーク | 何を捉えるか | AIエージェント時代に不足する点 |
|---|---|---|
| BPMN / Workflow Patterns | 処理の流れ・分岐・同期 | 責任主体の設計、AI導入後の変化の表現 |
| APQC PCF | 仕事の階層的分類 | 流れや依存関係の記述 |
| Value Stream Mapping | 価値の流れ・待ち・ボトルネック | 役割や責任の厳密な定義 |
| Service Blueprint | 表/裏の構造 | タスク間依存や責任割当 |
| Coordination Theory | 活動間の依存関係 | 時間軸の構造化 |
| Activity Theory / STS | 主体・道具・ルール・分業の関係 | ライフサイクルの段階設計 |

これらはそれぞれ有用だが、どれも単独では以下を同時に扱えない。

1. **時間軸の構造化** — 仕事がどの段階を経て進むか
2. **実行主体と責任主体の分離** — AIが実行しても責任は人に残るという構造
3. **裁量レベルの設計** — 「任せる度合い」の段階的な定義
4. **複数の観点からの可視化** — 流れ・依存・制御・測定を別々のビューで捉える

<Aside type="tip">
本モデルは、これらの既存フレームワークを否定するものではない。むしろ、それぞれの知見を統合的に使うための**共通の骨格**を提供する。
</Aside>

## このモデルが提供するもの

本モデルは、**テーラリング可能な参照モデル**である。特定の組織やツールの「正解の工程表」ではなく、ワークフローを記述・設計・変換するための共通フレームを提供する。

<CardGrid>
  <Card title="ライフサイクル（主軸）" icon="list-format">
    ソフトウェア開発の仕事の流れを9段階で構造化する。SDLCやDevOpsと整合しつつ、AIエージェントが入り込む粒度まで分解できる。
  </Card>
  <Card title="6つの設計属性" icon="setting">
    各段階に「誰が実行するか」「何を介して流れるか」「何が安全性を支えるか」「何を測るか」といった設計属性を重ねる。
  </Card>
  <Card title="8つのビュー" icon="open-book">
    単一の図では表現しきれない仕事の多面性を、目的別の8つのビューに分けて可視化する。
  </Card>

    モデルが固定するのは語彙と構造だけである。実行主体の割当、裁量レベルの選択、ゲート条件の設計はチーム・組織が自分たちのコンテキストに合わせて埋める。

## AIネイティブ化の基本認識

<Aside type="caution" title="前提">
AIネイティブなソフトウェアエンジニアリングは、開発ライフサイクルそのものを捨てるものではない。
</Aside>

既存のライフサイクルを背骨として保持したまま、各ステップにおける**実行主体、協調様式、制御環境、アーティファクトの設計を再編する**営みである。

従来は人が担っていた実行の一部にAIエージェントが入り込み、仕事は「人の逐次操作」から「意図・制約・評価基準を明示した委譲」へと変わる。その結果、エンジニアリングの中心は以下のように移動する。

| 従来の中心 | AIネイティブ化後の中心 |
|---|---|
| コードを書くこと | 仕事を安全かつ再現可能に進めるための設計 |
| 手順の実行 | 意図・制約・評価基準の明示 |
| 個人のスキル | ハーネス、ガードレール、観測可能性の設計 |

ただし、**責任主体は依然として人または組織に残る**。AIネイティブ化とは責任の放棄ではなく、**制約された裁量の再設計**である。

## このモデルが「ではないもの」

誤解を防ぐために、本モデルが何でないかを明示する。

- **特定の組織の工程表ではない** — 9段階のライフサイクルは参照形であり、そのまま採用することを前提としていない
- **AIの導入ガイドではない** — 「AIを入れるべきか」「どのツールが良いか」は答えない。AIを入れると決めた後の設計の枠組みを提供する
- **成熟度モデルではない** — レベル1〜5のような段階評価は行わない
- **完成品ではない** — 適用しながら育てる前提の参照モデルであり、フィードバックによって進化する
