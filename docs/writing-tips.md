# ドキュメント執筆 Tips

このモデルのドキュメントを書く・改善する過程で得られた教訓をまとめる。AIに書かせる場合にも、自分で書く場合にも適用できる。

---

## 用語選び

### 読み手が初見で意味を掴める語を使う

抽象的な用語は、書き手には明確でも読み手にはピンとこない。「これは何を指しているんだろう」と立ち止まらせない語を選ぶ。

- ✗ 設計属性 → ✓ ステップ構成（「各ステップの中身をどう組むか」がタイトルだけで伝わる）
- ✗ 設計判断を明示するための枠組み → ✓ 誰が何をどこまで担うかをワークフロー全体で設計するための枠組み（「何の設計か」を具体化する）

### モデルが提供するのは「方針・考え方」であり「機能」ではない

このモデルはツールではなく参照モデルである。「〜できる」という能力の提供ではなく、「〜する方針をとる」「〜という考え方で整理する」という表現が実態に合う。

- ✗ AIエージェントが入り込む粒度まで分解できる → ✓ AIエージェントが関与する粒度まで分解する方針をとる
- ✗ 複数の観点を統合的に可視化する → ✓ 複数の観点から可視化する方針をとる

### やったことを正確に表す動詞を使う

「組み直す」は「1つのものを分解して再構成する」を暗示するが、実際にやったのは「複数の既存知見を集めて1つの体系にまとめる」ことだった。動詞がやったことと合っているかを確認する。

- ✗ 組み直す → ✓ 体系化する

### 曖昧な短縮を避ける

省略すると何を指しているかわからなくなる場合がある。多少長くなっても、対象を明示する方が伝わる。

- ✗ AI前提の → ✓ AIエージェントを前提とした
- ✗ 骨格 → ✓ 枠組み（「骨格」は構造の一部、「枠組み」は全体の器というニュアンスの違い）

## 数値・定量表現

### 要素の個数を本文中に書かない

「9段階」「7要素」「6属性」「8ビュー」のような数値付き表現は、AI生成っぽさを出す。また、要素数が変わったときに全箇所を直す必要が生じる。

- ✗ 基底モデルの7要素 → ✓ 基底モデルの要素
- ✗ 全8ビューの定義を通読し → ✓ 各ビューの定義を通読し

定義表やリストの中で個数が自然に伝わるので、本文中で改めて数える必要はない。

## 文章の書き方

### 専門用語を導入前に使わない

読み手がまだ知らない概念を、説明なしにいきなり使うと理解を妨げる。導入ページでは特に注意する。

- ✗ 「暗黙知を明示する」「実行主体・制御環境・アーティファクトの設計」（導入1ページ目で）
- ✓ まず具体的な場面で説明してから、用語を導入する

### 表だけで終わらせない

表は情報を整理する手段としては優れるが、表だけのセクションは味気ない。文脈を伝える散文、具体例、図と組み合わせて初めて読み手に届く。

- ✗ 表1枚 + 一言の説明
- ✓ 散文で文脈を示す → 具体的な場面で説明 → 表やmermaid図でまとめる

### 既存の概念を新しいもののように見せない

「AI時代に加わる観点」として紹介したものが、実はAI以前から存在する概念だった場合、読み手の信頼を失う。何が本当に新しくて、何が「重要度が増した」だけかを区別する。

- ✗ 時間軸の構造化（AI時代に加わる観点として紹介）
- ✓ 仕事の分解の粒度が変わる（AIによって既存の概念の要求水準が上がることを正確に表現）

## ページ構成

### 1ページ1役割

複数の役割を1ページに持たせると、読み手が「このページは何を読むためのページか」を見失う。ページが長くなったり焦点がぼやけたら分割を検討する。

- model-goals = Why（なぜ必要か）
- model-overview = What（何であるか）
- what-you-can-do = How（どう使うか）
- reading-guide = Where（どこから読むか）

### ページ間で情報を重複させない

同じ内容が複数のページに書かれていると、更新漏れが起きる。情報は1箇所に集約し、他のページからはリンクで参照する。

- テーラリングの詳細 → model-overview に集約（what-you-can-do からは除去）
- ペルソナ別ルート → reading-guide に集約（index からはリンクで誘導）

### 補足的なセクションは本来の居場所に置く

「ではないもの」「対象外」のような補足セクションは、つい概要ページに入れたくなるが、そのページの主題からずれている場合は、内容に合ったページに集約する。概要ページからはリンクで導線をつなげば十分。

- ✗ model-overview（全体像）に「ではないもの」を置く → ページの役割がぼやける
- ✓ model-character（モデルの性格）に集約し、overview からはリンクで参照

### 見出し階層を正しく使う

h2の下にh3があるべきなのに、独立した概念をh3にしてしまうと、読み手が階層関係を誤解する。見出しレベルは内容の包含関係を反映させる。

## 図（Mermaid）

### 比較図は左右に並べる

「従来」と「導入後」の比較は、上下ではなく左右に並べると違いが一目で掴める。左に従来（As-Is）、右に導入後（To-Be）を配置する。

---

## 正確性

### 主張をソースと突き合わせる

「組み込んだ既存の知見」のような対応表を書くとき、実際の導出過程や参照元と突き合わせて正確かを確認する。書き手の印象で書くと、実際の由来とズレる。

- ✗ ステップ構成の由来: RACI, DORA（実際にはCoordination TheoryやVSMも由来に含まれる）
- ✗ ビューの由来: VSM（VSMの知見はビューの構造よりステップ構成の測定設計に効いている）

### 知見を正しい構成要素に帰属させる

ある知見がモデルのどの構成要素に影響しているかを正確に対応づける。VSMの知見がビューに影響しているのか、ステップ構成に影響しているのかで、読み手の理解が変わる。
