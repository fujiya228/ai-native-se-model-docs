## なぜ既存フレームワークを整理するか

AIエージェント前提でソフトウェアエンジニアリングのワークフローを再設計するには、「仕事の流れ」を言語化する共通の土台が必要になる。調査の結果、既存の説明体系は十分に豊富だが、それらは一つに統合されているわけではなく、異なる問いに答える複数の系統に分かれていることがわかった。

本モデルは単一のフレームワークに寄せるのではなく、以下の7つの系統をそれぞれの強みに応じて使い分ける。

## 7つの系統と本モデルでの使い方

| # | 系統 | 代表 | 本モデルでの使い方 |
|---|---|---|---|
| 1 | プロセス記述 | BPMN / Workflow Patterns | ライフサイクルの流れの記述 |
| 2 | 階層的分類 | APQC PCF | L0 / L1 / L2 の多段階粒度設計 |
| 3 | 価値流れ | Value Stream Mapping | フロー変数の定義、ボトルネック分析 |
| 4 | 表/裏構造 | Service Blueprint | レイヤービュー（Frontstage / Backstage / Support） |
| 5 | 協調・依存関係 | Coordination Theory | 依存関係ビューの4つの型 |
| 6 | 社会技術システム | Activity Theory / STS | Actor の定義、AIの多面性の理論的根拠 |
| 7 | 現実検証 | Process Mining | As-Is / To-Be / Validation ビュー |

## 各系統の詳細

### 1. BPMN / Workflow Patterns

| 項目 | 内容 |
|---|---|
| **強み** | 分岐、同期、例外、メッセージフローを表現できる。ライフサイクルの主流れや差し戻しの記述に有効 |
| **限界** | 責任の所在が弱い。価値流れや組織構造の説明には向かない |
| **本モデルでの使い方** | [ライフサイクルビュー](/views/view-1-lifecycle/)の基盤 |

### 2. APQC Process Classification Framework

| 項目 | 内容 |
|---|---|
| **強み** | 仕事を階層的に分類する地図。L0 / L1 / L2 のような多段階表現と相性が良い |
| **限界** | 流れそのものの記述には弱い。依存関係や詰まりは表現しにくい |
| **本モデルでの使い方** | [L2分解の原則](/appendix/l2-principles/)の設計参考 |

### 3. Value Stream Mapping（VSM）

| 項目 | 内容 |
|---|---|
| **強み** | ボトルネックの可視化、リードタイムや待ち時間の測定 |
| **限界** | 役割や責任の厳密な定義には向かない。制御環境の詳細設計には弱い |
| **本モデルでの使い方** | [フロー変数](/dynamics/flow-variables/)と[測定ビュー](/views/view-7-measurement/)の理論的基盤 |

### 4. Service Blueprint

| 項目 | 内容 |
|---|---|
| **強み** | 顧客に見える仕事と裏側の支援構造を分けて捉えられる |
| **限界** | タスク間依存や責任割当の厳密化には弱い |
| **本モデルでの使い方** | [レイヤービュー](/views/view-6-layer/)（Frontstage / Backstage / Support）の基盤 |

### 5. Coordination Theory（Malone & Crowston）

| 項目 | 内容 |
|---|---|
| **強み** | 仕事を「活動間の依存関係の管理」として捉える。依存の型ごとに調整コストが異なることを示す |
| **限界** | 時間軸上の流れや制御環境の記述は守備範囲外 |
| **本モデルでの使い方** | [依存関係ビュー](/views/view-4-dependency/)の4つの型、[動力学](/dynamics/theoretical-foundations/)の理論的基盤 |

### 6. Activity Theory / Sociotechnical Systems

| 項目 | 内容 |
|---|---|
| **強み** | 主体・対象・道具・ルール・共同体・分業の関係で仕事を捉える。AIが道具であり同時にルールを埋め込む装置でもあることを説明できる |
| **限界** | 実装レベルの詳細設計には向かない |
| **本モデルでの使い方** | [Actor の定義](/execution/actor-and-responsibility/)と[AIの4ファセット](/execution/ai-four-facets/)の理論的根拠 |

### 7. Process Mining

| 項目 | 内容 |
|---|---|
| **強み** | 設計図ではなく実際のログから流れを発見・検証する |
| **限界** | ログが取れない活動には適用できない |
| **本モデルでの使い方** | [As-Is / To-Be ビュー](/views/view-8-asis-tobe/)の Validation 面の発想 |

## この整理から得られた教訓

既存フレームワークの調査から、以下の4つの教訓が得られた。これらは本モデルの設計判断の根拠になっている。

| 教訓 | 本モデルへの反映 |
|---|---|
| 単一のフレームワークでは足りない | [複数ビュー](/views/why-multiple-views/)で投影する構造を採用 |
| ライフサイクルだけでは不十分 | ライフサイクルを主軸にしつつ、設計属性を重ねる構造を採用 |
| 要素だけでなく関係と動力学が必要 | [依存関係ビュー](/views/view-4-dependency/)と[動力学セクション](/dynamics/why-dynamics/)を追加 |
| 理想図だけでは不十分 | [As-Is / To-Be / Validation](/views/view-8-asis-tobe/) の3面構成を採用 |

<Aside>
これらの教訓の詳細な検討過程は[概念の導出過程](/appendix/derivation-process/)で解説している。
</Aside>
