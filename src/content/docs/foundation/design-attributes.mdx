## なぜ設計属性が必要か

ライフサイクルは「仕事がどの順序で流れるか」を示すが、それだけでは仕事を設計できない。各ステップについて「誰が進めるか」「どう協調するか」「何で安全を担保するか」といった設計判断が必要になる。

設計属性は、ライフサイクルの各ステップに**重ねて定義すべき6つの観点**を規定する。ライフサイクルが骨格だとすれば、設計属性はその骨格に肉付けをするための枠組みである。

## 6つの設計属性の一覧

| # | 属性名 | 英名 | 問い |
|---|---|---|---|
| 1 | 実行設計 | Execution Design | 誰がどの裁量で実行するか？ |
| 2 | 協調設計 | Collaboration Design | 主体間でどう連携するか？ |
| 3 | 制御環境 | Control Environment | 安全性・再現性をどう担保するか？ |
| 4 | アーティファクト | Artifact Design | 何を介して仕事を受け渡すか？ |
| 5 | 依存設計 | Dependency Design | ステップ間・チーム間の依存をどう管理するか？ |
| 6 | 測定設計 | Measurement Design | 何を測り、どう評価するか？ |

<Aside type="tip">
6属性は「何を定義すべきか」のチェックリストとして機能する。各属性の中身（誰を Responsible にするか、どの裁量レベルにするか等）はコンテキストに応じて埋める。
</Aside>

## 各属性の説明と使い方

### 1. 実行設計（Execution Design）

**問い:** 誰がどの裁量レベルでこのステップを実行するか？

実行主体（Actor）と責任主体の割り当て、および裁量レベル（L0〜L4）の選択を行う。AI ネイティブ化において最も直接的に変化する属性である。

**定義すべき項目:**
- RACI（Responsible / Accountable / Consulted / Informed）の割り当て
- 裁量レベルの選択（人が全て実行する L0 から、AI が自律実行する L4 まで）
- 実行主体と責任主体の分離の設計

**使い方の例:**
- Implementation の R を AI Agent に変更し、裁量レベルを L2（人が確認してから反映）に設定する
- Verification の自動検証部分は L3（AI が自律実行し、異常時のみ人に通知）に設定する

### 2. 協調設計（Collaboration Design）

**問い:** 複数の主体がどのように連携してこのステップを進めるか？

人と AI、人と人、AI と AI の間の協調方式を定義する。同期・非同期の選択、コミュニケーションの媒体、エスカレーションの経路が含まれる。

**定義すべき項目:**
- 協調の方式（ペアプログラミング、非同期レビュー、AI との対話等）
- 情報共有の手段とタイミング
- コンフリクト解消のルール

**使い方の例:**
- AI Agent が実装した変更を人がレビューする非同期フロー
- 複数の AI Agent が並列実装する際の作業領域の分離ルール

### 3. 制御環境（Control Environment）

**問い:** このステップの安全性・再現性・監査性をどう担保するか？

ゲート条件、自動検証、承認フロー、ロールバック手順など、仕事の品質と安全を支える仕組みの設計を行う。AI の裁量を広げるほど、この属性の重要度が増す。

**定義すべき項目:**
- ゲート条件（次のステップに進む条件）
- 自動検証の項目（CI、リンター、型チェック等）
- 承認フロー（誰が何を承認するか）
- 失敗時の対応（差し戻し、ロールバック）

**使い方の例:**
- AI が生成したコードは必ず自動テストを通過してからレビューに進む
- リスクの高い変更のみ人間の承認ゲートを設ける（リスクベースのゲート設計）

### 4. アーティファクト（Artifact Design）

**問い:** このステップの入力と出力として何を定義し、どのような形式で受け渡すか？

仕事の媒介物の設計。あるステップの出力が次のステップの入力となる受け渡しの質を高めるために、形式・粒度・品質基準を定義する。

**定義すべき項目:**
- 入力アーティファクトの一覧と品質基準
- 出力アーティファクトの一覧と品質基準
- フォーマットと粒度の規定

**使い方の例:**
- Design ステップの出力として、AI Agent が読み取れる構造化された設計書（マークダウン + コード例）を定義する
- Implementation の出力として、変更済みコード + テストコード + 変更理由の記述をセットで求める

### 5. 依存設計（Dependency Design）

**問い:** このステップは何に依存し、何がこのステップに依存しているか？

ステップ間の前後関係、並行実行可能な範囲、ブロッキングになりうる待ち合わせポイントを明示する。依存の設計が甘いと、AI の並列実行が生む恩恵をボトルネックが打ち消す。

**定義すべき項目:**
- 前提条件（開始に必要な入力や承認）
- 後続への影響（このステップの出力に依存するステップ）
- 並行実行の可否と条件

**使い方の例:**
- Implementation の複数タスクを並列に実行する際、互いの変更領域が重ならないことを前提条件として明記する
- Verification の自動検証と人レビューを並行にするか逐次にするかを設計する

### 6. 測定設計（Measurement Design）

**問い:** このステップの効率と品質をどの指標で評価するか？

仕事の流れの速度、品質、コストを評価するための指標を定義する。現状把握（記述モード）にも、目標設定（設計モード）にも、AI 導入前後の比較（変換モード）にも使える。

**定義すべき項目:**
- フロー指標（リードタイム、サイクルタイム、待ち時間）
- 品質指標（変更失敗率、差し戻し率、テストカバレッジ）
- コスト指標（トークン消費量、API コスト、人的工数）

**使い方の例:**
- Implementation の AI 導入前後でサイクルタイムと差し戻し率を比較する
- Verification のレビュー待ち時間を測定し、ゲート設計の改善に使う

## 設計属性とビューの関係

6つの設計属性は、それぞれ対応するビューによって可視化される。

| 設計属性 | 対応ビュー |
|---|---|
| 実行設計 | View 2: 実行設計ビュー |
| 協調設計 | View 2: 実行設計ビュー（協調様式の項目） |
| 制御環境 | View 5: 制御環境ビュー |
| アーティファクト | View 3: アーティファクトビュー |
| 依存設計 | View 4: 依存関係ビュー |
| 測定設計 | View 7: 測定ビュー |

<Aside>
ライフサイクルビュー（View 1）と As-Is / To-Be ビュー（View 8）は、特定の設計属性に対応するのではなく、すべての属性を時間軸や変換の観点から横断的に扱うビューである。
</Aside>
