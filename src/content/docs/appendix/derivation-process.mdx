## この文書の目的

最終形だけを提示するのではなく、途中の試行と棄却の理由を明示することで、モデルの設計根拠を第三者が検証・批判できるようにする。

## 出発点: 統合モデルの不在

[既存フレームワーク](/appendix/existing-frameworks/)の調査の結果、既存の説明体系は5つの系統に分断されていることがわかった。

| 系統 | 代表 | 強み | 不足 |
|---|---|---|---|
| 標準・規格 | ISO/IEC/IEEE 12207 | 語彙と厳密さ | 現場で共有しやすい図にならない |
| SDLC | IBM SDLC 等 | わかりやすさ | 継続運用・フィードバックループが薄い |
| DevOps / CD | DevOps infinity loop | 継続的改善 | 上流（発見・戦略・要件）が弱い |
| Value Stream | VSM, Team Topologies | 価値流れと組織責務 | 役割・制御の詳細は扱わない |
| AI時代の再定義 | McKinsey PDLC 等 | AI前提の役割再定義 | 統合されたモデルではない |

求めていた「一枚の統合モデル」は存在しなかった。

## 第一案: 3層モデル → 棄却

最初に3層で整理することを試みた。

1. **ライフサイクル層**: idea → discovery → ... → learn
2. **責務層**: human / AI agent / platform / reviewer / approver / operator
3. **制御層**: spec / context / guardrails / CI/CD / observability

**棄却理由**: 責務層が異質な概念を混在させていた。実行主体（human / AI agent）、役割（reviewer / approver）、実行の流し方（workflow）、基盤（platform）が1つの層に混ざり、分類軸が衝突した。

## 第二案: 4軸モデル → 7つの批判で転換

概念の型を揃えた4軸で再構成した。

- 軸1: ライフサイクル、軸2: 主体・役割、軸3: 実行様式、軸4: 制御・基盤

この4軸モデルに対して、6つの研究伝統の視点から7つの批判的指摘を受けた。

| # | 指摘 | 影響 |
|---|---|---|
| 1 | 概念の型が揃っていない | ライフサイクルは時間軸、残りは設計変数。同列に並べるのは不自然 |
| 2 | AIに「責任を持たせる」のは危険 | 実行主体と責任主体を分ける必要がある |
| 3 | ガードレールを「AIの補助」と捉えると狭すぎる | 仕事全体を成立させる制御環境として位置づける |
| 4 | workflowは広すぎる | 協調様式として再定義する |
| 5 | アーティファクトが欠けている | 仕事はアーティファクトを介した非同期受け渡しで流れる |
| 6 | 記述モデルと処方モデルが混ざっている | As-Is / To-Be / Validation の3面が必要 |
| 7 | ステップ間の接続不良が見えない | 受け渡し条件・差し戻し条件・観測点の定義が必要 |

## 転換: 主軸 + 設計属性への再編

7つの指摘を受けて、「4軸が全部同じレベルで並ぶ」という前提を捨てた。

- **ライフサイクルは背骨**（主軸）
- **残りはステップ属性**（各ステップに重ねる設計変数）
- **AI-native化は属性の再設計**

### 設計属性の確定

4軸から6つの設計属性に再編した。

| # | 設計属性 | 由来 | 追加理由 |
|---|---|---|---|
| 1 | 実行設計 | 4軸の軸2から | — |
| 2 | 協調設計 | 4軸の軸3を再定義 | — |
| 3 | 制御環境 | 4軸の軸4から | — |
| 4 | アーティファクト | 批判5で追加 | 非同期受け渡しの記述に不可欠 |
| 5 | 依存設計 | Coordination Theory の要請 | ステップ間の関係が見えないと、ボトルネック移動が予測できない |
| 6 | 測定設計 | VSM / DORA の要請 | 効果も副作用も評価できない |

## 基底モデル: 7要素の導出

設計属性だけではステップ間の関係が落ちるという問題に対し、共通の基底モデルを複数ビューで投影する形に移行した。[基底モデルの7要素](/foundation/base-model/)として、Object / Lifecycle Step / Actor / Artifact / Dependency / Control / Metric を定義した。

## 複数ビューへの移行

マトリクス（ステップ x 属性）だけでは、6つの研究伝統が重視する「関係」を表現できないという矛盾を解消するために、[8つのビュー](/views/overview/)を設計した。

## 設計判断の要約

| 判断 | 棄却した案 | 理由 |
|---|---|---|
| ライフサイクルを主軸にする | 4軸を同列に並べる | 概念の型が揃わない |
| 設計属性を6つにする | 4属性 | 依存関係と測定が欠落するとボトルネック移動と効果検証ができない |
| アーティファクトを独立させる | 制御環境の一部として扱う | AI-nativeの仕事はアーティファクトを介した非同期受け渡しが中心 |
| 複数ビューで投影する | 単一のマトリクス | 関係がマトリクスでは表現できない |
| Actor の境界を「自律的に仕事を前進させるか」で引く | AI / ツールの二分法 | 自動化も Actor に入る。基準は行為能力 |
| 責任主体を実行主体と分ける | AIに責任を持たせる | 法的・組織的に責任は人/組織に残る |
| L2は差が生じるところだけ詳細化する | 全L1を均等にL2分解 | 差が出ないL2は設計判断を増やさない |
| モデルを不変層/設計変数層/実装層に分ける | AIの配置をモデルに固定する | 配置は技術・組織の成熟度で変わる変数 |

<Aside>
これらの設計判断は「正解」ではなく「試行と批判の結果」である。異なるコンテキストでは異なる判断が妥当でありうる。導出過程を公開するのは、批判と改善を可能にするためである。
</Aside>
