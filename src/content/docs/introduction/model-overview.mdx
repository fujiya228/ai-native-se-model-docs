## 既存の知見を、AIエージェントを前提とした枠組みに組み直す

本モデルは、既存の枠組みの知見に[AIエージェントがもたらす変化](/introduction/model-goals/#aiエージェントによって変わること)への対応を加え、**AIエージェントを前提としたワークフロー設計の枠組みとして組み直した参照モデル**である。

| 本モデルの構成要素 | 目的 | 組み込んだ既存の知見 | 新たに加えた観点 |
|---|---|---|---|
| ライフサイクル | 開発全体の段階を定義する | SDLC, ISO 12207 | AI前提の粒度分解（L2） |
| ステップ構成 | 各段階の中身を整理する | RACI, VSM, DORA, Coordination Theory | 実行主体/責任主体の分離、制御環境・測定を各段階に重ねる |
| ビュー | 複数の観点から可視化する | 4+1 View Model, Team Topologies | 共通の基底モデルから目的別に投影 |
| テーラリング | チームのコンテキストに合わせる | ISO 12207 | 語彙と構造だけを固定し、中身は埋める |

<Aside type="tip">
既存フレームワークの種別整理、個別の強み・限界、および本モデルとの詳細な比較は[既存フレームワークの全体像](/appendix/existing-landscape/)で解説している。
</Aside>

## このモデルが提供するもの

本モデルは、**テーラリング可能な参照モデル**である。特定の組織やツールの「正解の工程表」ではなく、ワークフローを記述・設計・変換するための共通フレームを提供する。

<CardGrid>
  <Card title="ライフサイクル（主軸）" icon="list-format">
    ソフトウェア開発の仕事の流れを構造化する。SDLCやDevOpsと整合しつつ、AIエージェントが入り込む粒度まで分解できる。
  </Card>
  <Card title="ステップ構成" icon="setting">
    各段階に「誰が実行するか」「何を介して流れるか」「何が安全性を支えるか」「何を測るか」といったステップ構成を重ねる。
  </Card>
  <Card title="ビュー" icon="open-book">
    単一の図では表現しきれない仕事の多面性を、目的別のビューに分けて可視化する。
  </Card>
  <Card title="テーラリング" icon="pencil">
    モデルが固定するのは語彙と構造だけである。中身はチーム・組織が自分たちのコンテキストに合わせて埋める。
  </Card>
</CardGrid>

## テーラリング: 自分たちのコンテキストに合わせる

本モデルは「正解の工程表」を提供するものではない。モデルが固定するのは**語彙と構造**（ライフサイクルの骨格、基底要素、ステップ構成、ビュー）であり、以下はすべてチーム・組織が自分たちのコンテキストに合わせて埋める**設計変数**である。

| 設計変数 | 例 |
|---|---|
| 実行主体の割当 | ImplementationのResponsibleをAI Agentにするか、Humanにするか |
| 裁量レベルの選択 | L2（計画に沿って自律実行）か、L3（目標を与えれば計画から自律）か |
| RACIの中身 | 誰がAccountableで誰がConsultedか |
| ゲート条件の厳しさ | 全PR人間レビュー必須か、リスクベースで判断するか |
| 測定指標の具体値 | リードタイムの目標値をいくつにするか |
| 使用ツール・基盤 | Claude CodeかCodex CLIか、どのCI/CDを使うか |

<Aside type="note">
テーラリングの最小セットは以下の3つである。(1) ライフサイクルビューで対象段階の開始・終了条件を定義する、(2) 実行設計ビューでRACIと裁量レベルを埋める、(3) 依存関係ビューで主要な依存を特定する。この3点があれば「誰が何をどの順序で進め、どこで待ちが生じるか」が見える。
</Aside>

## このモデルが「ではないもの」

誤解を防ぐために、本モデルが何でないかを明示する。

- **特定の組織の工程表ではない** — ライフサイクルは参照形であり、そのまま採用することを前提としていない
- **AIの導入ガイドではない** — 「AIを入れるべきか」「どのツールが良いか」は答えない。AIを入れると決めた後の設計の枠組みを提供する
- **成熟度モデルではない** — レベル1〜5のような段階評価は行わない
- **完成品ではない** — 適用しながら育てる前提の参照モデルであり、フィードバックによって進化する
