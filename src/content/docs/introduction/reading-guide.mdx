## 全体構成

本サイトは以下のセクションで構成されている。

| セクション | 内容 | 読む目安 |
|---|---|---|
| **導入** | モデルの必要性、できること、読み方 | 全員 |
| **基礎概念** | 基底モデル、ステップ構成、モデルの性格 | 全員（最低限ここまで） |
| **ライフサイクル** | ライフサイクルの定義、L2分解、依存関係 | 設計・適用する人 |
| **実行設計** | 実行主体、責任主体、RACI、裁量レベル | 設計・適用する人 |
| **ビュー体系** | ビューの定義と使い方 | 深く理解したい人 |
| **動力学** | フロー変数、ボトルネック移動、WIP制限 | 運用・改善に関わる人 |
| **適用例** | モデルを特定コンテキストに適用した実例 | 実践したい人 |
| **付録** | 既存フレームワーク、導出過程、用語集 | 背景を知りたい人 |

<Aside type="tip">
全部を順番に読む必要はない。まず核だけ掴みたいなら次の最短ルート、役割や関心が明確なら後半のペルソナ別ルートから始めるとよい。
</Aside>

## 中核概念への最短ルート

本モデルはライフサイクル・ビュー体系・動力学と多層の構成を持つが、既存のプロセスモデルとの違いを生んでいる考え方は少数である。構成の全体を覚える前に、まずその核だけを掴むための読み順を示す。

<Card title="最短ルート">
1. [このモデルが目指すこと](/introduction/model-goals/) — どんな問題を解こうとしているかを押さえる
2. [実行主体と責任主体](/execution/actor-and-responsibility/) — 「誰が実行するか」と「誰が責任を持つか」を分ける。AIが実行を担うと両者の一致が崩れる、という本モデルの出発点
3. [RACIと裁量レベル](/execution/raci-and-discretion/) — 「どこまでAIに任せるか」を段階として設計する道具
4. [なぜ動力学が必要か](/dynamics/why-dynamics/) — 個々のステップではなく、フロー全体で変化を捉える視点
5. [ボトルネック移動の4パターン](/dynamics/bottleneck-patterns/) — AI導入で速くなった先で、どこが新たに詰まるかの予測
</Card>

**読んだ後にできること**: 「誰に何をどこまで任せ、その結果ワークフロー全体がどう変わるか」という本モデルの中心の問いを、自分の言葉で説明できる。

これ以外の領域は、この核を実際のワークフローに展開するときに読む深掘りであり、最初に読む必要はない。

- ワークフロー全体を段階に分けて整理したくなったら — [ライフサイクル](/lifecycle/nine-stages/)と[L2分解表](/lifecycle/l2-decomposition/)
- 設計した内容を漏れなく記述・共有したくなったら — [ビュー体系](/views/why-multiple-views/)
- モデルの背景や既存フレームワークとの関係を知りたくなったら — [付録](/appendix/existing-landscape/)

## ペルソナ別の推奨ルート

### エンジニア — 「AIエージェントを自分の開発に取り入れたい」

日常の開発作業でAIエージェントをどう活用するか、どこまで任せてどこで自分が判断するかを知りたい人向け。

<Card title="推奨ルート">
1. **導入** — [このモデルが目指すこと](/introduction/model-goals/) で全体像を把握
2. **基礎概念** — [基底モデル](/foundation/base-model/) で語彙を押さえる
3. **実行設計** — [RACIと裁量レベル](/execution/raci-and-discretion/) で「どこまで任せるか」の設計方法を理解する
4. **適用例** — [Implementation: 変換モード](/instances/implementation/) で具体的なAI導入設計を見る
</Card>

**読んだ後にできること**: 自分の開発タスクに対して「ここはAIに任せる / ここは自分が判断する」を構造的に整理できる。

---

### テックリード — 「チームの開発プロセスをAIエージェントを前提に再設計したい」

チームの開発ワークフロー全体を見渡し、AIエージェント導入の計画を立てたい人向け。

<Card title="推奨ルート">
1. **導入** — [このモデルが目指すこと](/introduction/model-goals/) と [このモデルの使い方](/introduction/how-to-use/) で全体像を把握
2. **基礎概念** — [ステップ構成](/foundation/design-attributes/) で各段階に何を定義すべきかを理解する
3. **ライフサイクル** — [ライフサイクル](/lifecycle/nine-stages/) と [依存関係マップ](/lifecycle/dependency-map/) でワークフロー全体を構造化する
4. **ビュー体系** — [View 8: As-Is / To-Be / Validation](/views/view-8-asis-tobe/) でAI導入前後の差分設計の方法を学ぶ
5. **動力学** — [ボトルネック移動の4パターン](/dynamics/bottleneck-patterns/) でAI導入後に起きるボトルネック移動を理解する
6. **適用例** — 具体的なインスタンスを参考にする
</Card>

**読んだ後にできること**: チームの開発プロセスをAs-Is/To-Beで整理し、AIエージェント導入の段階的な計画を策定できる。

---

### ツール開発者 — 「開発支援ツールの設計に活かしたい」

AIエージェントのオーケストレーター、開発支援ツール、CI/CDパイプラインなどを設計・開発する人向け。

<Card title="推奨ルート">
1. **導入** — [このモデルが目指すこと](/introduction/model-goals/) で問題設定を把握
2. **基礎概念** — [基底モデル](/foundation/base-model/) の要素と [モデルの性格](/foundation/model-character/) でモデルの設計原則を理解する
3. **実行設計** — [実行主体と責任主体](/execution/actor-and-responsibility/) と [AIの多面性](/execution/ai-four-facets/) でAIの位置づけを押さえる
4. **ビュー体系** — [View 5: 制御環境ビュー](/views/view-5-control/) と [View 6: レイヤービュー](/views/view-6-layer/) でツールが担うべき制御と構造を理解する
5. **ライフサイクル** — [L2分解表](/lifecycle/l2-decomposition/) でツールがカバーすべき粒度を把握する
</Card>

**読んだ後にできること**: 自分のツールがライフサイクルのどの段階・どのビューに対応するかを明確にし、不足している機能や制御を特定できる。

---

### 研究者 — 「AIネイティブSEの概念的枠組みを理解したい」

AIネイティブなソフトウェアエンジニアリングの理論的な基盤や、既存研究との関係を知りたい人向け。

<Card title="推奨ルート">
1. **導入** — [このモデルが目指すこと](/introduction/model-goals/) で問題設定と既存フレームワークの位置づけを把握
2. **基礎概念** — 全ページを順に読み、モデルの構造と設計原則を理解する
3. **付録** — [既存フレームワークの全体像](/appendix/existing-landscape/) で BPMN, VSM, Coordination Theory 等との関係を確認する
4. **付録** — [概念の導出過程](/appendix/derivation-process/) でモデルがどう導出されたかの過程を追う
5. **ビュー体系** — 各ビューの定義を通読し、モデルの表現力を評価する
6. **動力学** — [理論的基盤](/dynamics/theoretical-foundations/) でToC, リトルの法則等との接続を確認する
7. **付録** — [自己批評と未解決論点](/appendix/critique-and-gaps/) で現時点の限界を把握する
</Card>

**読んだ後にできること**: 本モデルの理論的な位置づけを理解し、自身の研究で参照・拡張・批判するための基盤を得られる。
