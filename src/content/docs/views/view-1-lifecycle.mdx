## 目的

仕事の**時間的な流れ**を表す。ステップの順序、ループ、差し戻しを可視化し、全体の進行構造を明らかにする。

このビューは「誰がやるか」「何で測るか」には答えない。それらは実行設計ビューや測定ビューの役割である。

## メタ定義

| メタ項目 | 定義 |
|---|---|
| **目的** | 仕事の時間的な流れを表す。ステップの順序、ループ、差し戻しを可視化する |
| **主語** | L1 / L2 ステップ |
| **最小記述単位** | L1（粗め扱い）または L2（詳細扱い） |
| **記述項目** | ステップ名、目的、開始条件、終了条件、入出力アーティファクト、差し戻し先 |
| **停止基準** | 各ステップの開始条件・終了条件・差し戻し先が定義されていれば十分 |
| **ライフサイクルとの対応** | このビュー自体がライフサイクルの正本 |
| **いつ使うか** | 全体の流れを俯瞰するとき。他のビューの基盤として常に参照される |

## 表現するもの

ライフサイクルビューは以下の構造を表現する。

### 主要ステップ

[L1 9段階](/lifecycle/nine-stages/)として定義された、Discovery から Learn までの9つのステップ。

### 主ループ

ソフトウェア開発の中核的な繰り返し。

- **Implementation → Verification & Review → Release** — 実装・検証・リリースのサイクル
- **短期フィードバック** — Verification での差し戻しによる Implementation への戻り

### 差し戻し

品質ゲートを通過しなかったときの後退パス。

| 差し戻し元 | 差し戻し先 | 条件 |
|---|---|---|
| Verification & Review | Implementation | コード品質の問題 |
| Verification & Review | Design | 設計方針の問題 |
| Verification & Review | Specification | 要件の不備・曖昧さ |

### 長期フィードバック

- **Learn → Discovery** — 運用から得た知見が新しい発見に繋がる
- **Monitoring & Ops → Framing** — 本番の問題が新しい問題定義を生む

## 他のビューとの関係

ライフサイクルビューは全ビューの**背骨**として機能する。

| ビュー | ライフサイクルビューとの関係 |
|---|---|
| [実行設計ビュー](/views/view-2-execution/) | 各ステップに RACI を重ねる |
| [アーティファクトビュー](/views/view-3-artifact/) | 入出力アーティファクトの詳細を記述する |
| [依存関係ビュー](/views/view-4-dependency/) | ステップ間の矢印を構造化する |
| [制御環境ビュー](/views/view-5-control/) | 各ステップの安全装置を記述する |
| [測定ビュー](/views/view-7-measurement/) | 各ステップに指標を紐づける |

<Aside>
ライフサイクルビューは「他のビューの下敷き」であるが、ライフサイクル自体の詳細は[ライフサイクルセクション](/lifecycle/nine-stages/)で解説している。このページではビューとしての位置づけに焦点を当てる。
</Aside>

## AIネイティブ文脈での変化

ライフサイクルの構造自体はAI導入によって変わらない。変わるのは、各ステップの**中身**と**速度**、そしてステップ間の**バランス**である。

- **速度の変化** — Implementation が高速化し、Verification がボトルネックになる
- **ループの頻度** — 差し戻しと再実装のサイクルが短くなる
- **新しいパスの出現** — 低リスク変更の自動マージにより、一部のステップをスキップするパスが生まれうる

これらの変化はライフサイクルビュー自体には表れにくく、[依存関係ビュー](/views/view-4-dependency/)や[測定ビュー](/views/view-7-measurement/)を併用して初めて可視化できる。
