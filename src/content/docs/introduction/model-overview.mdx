## 構成要素

本モデルは、既存の枠組みの知見に[AIエージェントがもたらす変化](/introduction/model-goals/#aiエージェントによって変わること)への対応を加え、**AIエージェントを前提としたワークフロー設計の枠組みとして体系化した参照モデル**である。

<Card title="ライフサイクル（主軸）" icon="list-format">
  ソフトウェア開発の仕事の流れをステップとして整理する。SDLCやDevOpsと整合する参照形であり、AIエージェントが関与する粒度まで分解する方針をとる。
</Card>

<Card title="ステップ構成" icon="setting">
  各ステップに「誰が実行するか」「何を介して流れるか」「何が安全性を支えるか」「何を測るか」といった構成を重ねる。
</Card>

<Card title="ビュー" icon="open-book">
  単一の図では表現しきれない仕事の多面性を、目的別のビューに分けて可視化する。
</Card>

<Aside type="tip">
各構成要素が組み込んだ既存の知見（SDLC, RACI, DORA 等）との対応は[既存フレームワークの全体像](/appendix/existing-landscape/)で解説している。
</Aside>

## テーラリング: 自分たちのコンテキストに合わせる

テーラリングとは、参照モデルを自分たちのコンテキスト（チーム規模、プロダクト特性、リスク許容度など）に合わせて調整することである。本モデルはこのテーラリングを前提として設計されている。

モデルが固定するのは**語彙と構造**（ライフサイクルの枠組み、基底要素、ステップ構成、ビュー）であり、各ステップ構成の中身はチーム・組織が自分たちのコンテキストに合わせて埋める。

下表は[ステップ構成](/foundation/design-attributes/)の6観点ごとに、テーラリングで検討しうる項目を例示したものである。網羅的な一覧ではなく、各観点の問いに答える過程で、自チーム固有の調整項目が現れることを想定している。

| ステップ構成 | 調整の例（一例） |
|---|---|
| 実行設計 | 誰が実行するか／AIにどこまで裁量を委ねるか（L0〜L4）／責任主体を誰にするか |
| 協調設計 | 人とAIの連携方式（対話・レビュー・ペア）／コンフリクト解消のルール |
| 制御環境 | 次のステップに進むゲート条件／自動検証の範囲／承認フロー |
| アーティファクト | 入出力の形式と粒度／AIが読み書きしやすい構造の採用可否 |
| 依存設計 | 並行実行の可否と条件／待ち合わせポイントの置き方 |
| 測定設計 | リードタイム・変更失敗率など指標の具体値／測定の頻度 |

<Aside type="note">
上記はあくまで例であり、実際の調整項目はプロダクト特性やチームのリスク許容度によって変わる。ステップ構成は「何を定義すべきか」のチェックリストとして機能するので、各観点の問いに答えていけば、自チーム固有の調整項目が見えてくる。
</Aside>
