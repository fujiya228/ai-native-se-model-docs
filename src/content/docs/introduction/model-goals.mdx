## AIエージェント時代の問い

AIエージェントがコードを書き、テストを実行し、PRを作成する。この変化はすでに起きている。しかし、多くのチームが直面しているのは「どのツールを使うか」ではなく、もっと根本的な問いである。

- 仕事の流れのどこにAIを入れるのか
- どこまでの裁量を委ね、どこに人の判断を残すのか
- 安全に回すために何を設計しておくべきか
- AI導入前後で何が変わり、何が変わらないのか

これらの問いに答えるには、「ツールの機能比較」でも「AIに全部任せる/任せない」の二択でもなく、**ワークフロー全体を構造的に捉え、設計判断を明示するための枠組み**が必要になる。

## 仕事の流れを整理する枠組みは、すでに豊富にある

仕事の流れを回す・捉える・改善するための枠組みは、すでに多数存在する。それらは目的ごとに使い分けるものであり、実務では複数を組み合わせて使うのが当たり前である。

### 目的別の整理

**汎用の枠組み** — 業界を問わず使えるもの。

| 目的 | 代表例 | 何をするか |
|---|---|---|
| 改善を回す | PDCA, OODA, DMAIC | 計画→検証の反復、高速な判断 |
| 流れを図にする | フローチャート, BPMN | 手順・分岐・担当の可視化 |
| 全体を俯瞰する | SIPOC, Value Stream Mapping | 入出力の整理、ボトルネックの特定 |
| 役割を明確にする | RACI | 実行・承認・相談・共有の割当 |
| 作業を分解・管理する | WBS, ガントチャート, カンバン | タスク分解、スケジュール、進捗管理 |

**エンジニアリング特化の枠組み** — ソフトウェア開発の「どう進めるか」に踏み込むもの。

| 目的 | 代表例 | 何をするか |
|---|---|---|
| 開発全体の段階を定義する | SDLC, ISO/IEC/IEEE 12207 | ライフサイクルの共通語彙と標準化 |
| チームの進め方を決める | Scrum, Kanban | 反復・フロー管理 |
| 開発→運用を繋ぐ | DevOps, CI/CD | 連続的なデリバリーの自動化 |
| デリバリーの健全性を測る | DORA, SPACE | 速度・安定性・開発者体験の定量評価 |
| チーム構造を設計する | Team Topologies | 認知負荷とフローの関係 |

実務ではこれらを組み合わせて使う。たとえば「SIPOC で全体像を掴み → フローチャートで手順を描き → RACI で役割を決め → カンバンで進捗を管理し → PDCA で改善を回す」といった具合である。

### AI時代に加わる観点

人やシステムに加えて、AIエージェントという少し異なる性質を持つ実行主体が現れた。自律的に判断し、コードを書き、タスクを完了する——しかし責任は取れない。この新しい実行主体をワークフローに組み込むために、以下の観点を加える必要がある。

| 観点 | なぜ必要か |
|---|---|
| **時間軸の構造化** | AIが入り込む粒度まで仕事の段階を分解する必要がある |
| **実行主体と責任主体の分離** | AIが実行しても、責任は人に残る。この構造を設計する必要がある |
| **裁量レベルの設計** | 「全部任せる/任せない」ではなく、段階的に委譲度を設計する必要がある |
| **複数の観点からの統合的な可視化** | 流れ・依存・制御・測定を統合的に捉える必要がある |

### 既存の知見を、AI時代の骨格に組み直す

本モデルは、既存の枠組みの知見にこれらの観点を加え、**AI前提のワークフロー設計に必要な骨格として組み直した参照モデル**である。

| 本モデルの構成要素 | 目的 | 組み込んだ既存の知見 | 新たに加えた観点 |
|---|---|---|---|
| 9段階ライフサイクル | 開発全体の段階を定義する | SDLC, ISO 12207 | AI前提の粒度分解（L2） |
| 6つの設計属性 | 各段階で何を設計すべきかを示す | RACI, DORA | 実行主体/責任主体の分離、制御環境・測定を各段階に重ねる |
| 8つのビュー | 複数の観点を統合的に可視化する | VSM, Team Topologies | 共通の基底モデルから目的別に投影 |
| テーラリング | チームのコンテキストに合わせる | ISO 12207 | 語彙と構造だけを固定し、中身は埋める |

本モデルは、**テーラリング可能な参照モデル**である。特定の組織やツールの「正解の工程表」ではなく、ワークフローを記述・設計・変換するための共通フレームを提供する。

## AIネイティブ化の基本認識

<Aside type="caution" title="前提">
AIネイティブなソフトウェアエンジニアリングは、開発ライフサイクルそのものを捨てるものではない。
</Aside>

既存のライフサイクルを背骨として保持したまま、各ステップにおける**実行主体、協調様式、制御環境、アーティファクトの設計を再編する**営みである。

従来は人が担っていた実行の一部にAIエージェントが入り込み、仕事は「人の逐次操作」から「意図・制約・評価基準を明示した委譲」へと変わる。その結果、エンジニアリングの中心は以下のように移動する。

| 従来の中心 | AIネイティブ化後の中心 |
|---|---|
| コードを書くこと | 仕事を安全かつ再現可能に進めるための設計 |
| 手順の実行 | 意図・制約・評価基準の明示 |
| 個人のスキル | ハーネス、ガードレール、観測可能性の設計 |

ただし、**責任主体は依然として人または組織に残る**。AIネイティブ化とは責任の放棄ではなく、**制約された裁量の再設計**である。

## このモデルが「ではないもの」

誤解を防ぐために、本モデルが何でないかを明示する。

- **特定の組織の工程表ではない** — 9段階のライフサイクルは参照形であり、そのまま採用することを前提としていない
- **AIの導入ガイドではない** — 「AIを入れるべきか」「どのツールが良いか」は答えない。AIを入れると決めた後の設計の枠組みを提供する
- **成熟度モデルではない** — レベル1〜5のような段階評価は行わない
- **完成品ではない** — 適用しながら育てる前提の参照モデルであり、フィードバックによって進化する
