## このページの目的

[このモデルが目指すこと](/introduction/model-goals/)では、既存の枠組みを目的ごとに組み合わせて使うのが実務の当たり前であること、そしてAI時代にはそこに新たな観点を加えて組み直す必要があることを述べた。

このページでは、その背景を詳しく見ていく。既存の枠組みを**目的と種別ごとに整理**し、それぞれが**何を捉え、何を捉えないか**を明らかにした上で、本モデルがどの知見をどう組み込んでいるかを示す。

---

## 枠組みの種別

「仕事の流れを捉えるフレームワーク」と一口に言っても、目的と手段は大きく異なる。以下の種別に分けることで、混同を避けられる。

| 種別 | 目的 | 代表例 |
|---|---|---|
| **改善サイクル** | 改善を反復的に回す | PDCA, OODA, DMAIC |
| **プロセス記法** | プロセスを図として記述する | BPMN, Workflow Patterns |
| **可視化・改善手法** | 流れの中の無駄やボトルネックを特定する | Value Stream Mapping, SIPOC |
| **分類体系** | 業務を階層的に分類する | APQC PCF |
| **設計手法** | 顧客接点と裏側プロセスを構造化する | Service Blueprint |
| **発見手法** | ログから実際の流れを再構成する | Process Mining |
| **分析理論** | 協調や活動を学術的に分析する | Coordination Theory, Activity Theory / STS |
| **ライフサイクル定義** | 仕事がどの段階を経て進むかを定義する | SDLC, ISO/IEC/IEEE 12207, V-model |
| **チーム運営** | 開発の進め方・反復・フロー管理を定義する | Scrum, Kanban, XP |
| **デリバリーモデル** | 開発から運用までの連続的な流れを設計する | DevOps, CI/CD, GitHub Flow |
| **ゲートモデル** | ステージごとの意思決定ゲートで制御する | Stage-Gate |
| **測定** | フローの健全性を定量的に評価する | DORA, SPACE, DevEx |
| **組織設計** | チーム構造とフローの関係を設計する | Team Topologies |
| **価値接続** | エンジニアリング指標とビジネス成果を接続する | Flow Framework |

<Aside>
この種別分類自体が本調査の成果の一つである。既存の文献では、これらを一律に「フレームワーク」と呼ぶことが多いが、目的と使い方が根本的に異なるため、混同すると議論が噛み合わなくなる。
</Aside>

## 汎用の枠組み

業界を問わず、仕事の流れを回す・捉える・改善するためのもの。ソフトウェアエンジニアリングでも利用可能だが、開発固有の課題（ライフサイクル、テスト、デプロイ等）には踏み込まない。

### 改善サイクル: PDCA / OODA / DMAIC

**PDCA**（Plan-Do-Check-Act）は Deming が日本に紹介した改善サイクルで、業務改善の基本形として最も広く知られている。計画を立て、実行し、結果を検証し、改善する——この反復が仕事の質を上げるという考え方である。

**OODA**（Observe-Orient-Decide-Act）は米空軍のジョン・ボイドが提唱した意思決定ループ。PDCAが「計画→検証」を重視するのに対し、OODAは「観察→状況判断」に重きを置き、高速な意思決定に適する。

**DMAIC**（Define-Measure-Analyze-Improve-Control）は Six Sigma の中核手法。データ駆動で厳密な改善を回す。

| 枠組み | 特徴 | 強み | 限界 |
|---|---|---|---|
| PDCA | 計画→実行→検証→改善の反復 | シンプルで汎用的。あらゆる業務に適用可能 | 流れそのものの記述は目的外。抽象的すぎて形骸化しやすい |
| OODA | 観察→状況判断→意思決定→実行 | 不確実な状況での高速な行動に向く | 品質管理や標準化には弱い |
| DMAIC | Define→Measure→Analyze→Improve→Control | データ駆動の厳密な改善手法 | 導入コストが高い。定量化困難な業務には適用しにくい |

これらは「仕事をどう回すか」の基本原則を提供するが、仕事の**構造**（どの段階で何をするか、誰が担うか）は扱わない。

### プロセス記法: BPMN / Workflow Patterns

OMG（Object Management Group）が管理する国際標準（ISO 19510）。業務プロセスの流れ、分岐、同期、イベントを厳格な記法で表現する。

| 強み | 限界 |
|---|---|
| 標準記法として関係者間で共有しやすい | 責任の所在が弱い |
| As-Is / To-Be の業務フロー記述に有効 | AI導入後の変化を直接表現する枠組みではない |
| システム実装や自動化に耐える精度 | 組織構造や価値流れの説明には向かない |

**本モデルでの使い方:** [ライフサイクルビュー](/views/view-1-lifecycle/)の基盤。分岐、同期、例外、メッセージフローの表現に活用。

### 可視化手法: Value Stream Mapping / SIPOC

**Value Stream Mapping**（VSM）はトヨタ生産方式（TPS）由来。工程と待ち時間を可視化し、リードタイムとサイクルタイムの比率からボトルネックを特定する。

**SIPOC**（Supplier-Input-Process-Output-Customer）は、プロセスの入出力と関係者を一覧で整理する手法。全体像を俯瞰するのに適する。

| 枠組み | 強み | 限界 |
|---|---|---|
| VSM | 「流れ」だけでなく「停滞」も見える。ソフトウェア分野では Value Stream Management に発展 | 役割や責任の厳密な定義は扱わない。制御環境の詳細設計は扱わない |
| SIPOC | プロセスの入出力を一覧で整理できる。新メンバーへの全体説明に有効 | 流れの詳細や分岐は扱わない。改善策の導出には弱い |

**本モデルでの使い方:** [フロー変数](/dynamics/flow-variables/)と[測定ビュー](/views/view-7-measurement/)の理論的基盤。ボトルネックの可視化、リードタイムや待ち時間の測定に活用。

### 設計手法: Service Blueprint

顧客に見える仕事（フロントステージ）と裏側の支援構造（バックステージ）を分けて可視化する。サービスデザイン領域で広く使われる。

| 強み | 限界 |
|---|---|
| 顧客視点とオペレーションを一枚で見やすい | タスク間依存や責任割当の厳密化には弱い |
| 表/裏構造の区別が明確 | 純粋な内部開発フローの管理にはやや不向き |

**本モデルでの使い方:** [レイヤービュー](/views/view-6-layer/)（Frontstage / Backstage / Support）の基盤。

### その他の汎用枠組み

| 枠組み | 種別 | 概要 | 本モデルでの使い方 |
|---|---|---|---|
| APQC PCF | 分類体系 | 業務プロセスの階層的分類 | [L2分解の原則](/appendix/l2-principles/)の設計参考。L0 / L1 / L2 の多段階粒度設計に活用 |
| Process Mining | 発見手法 | イベントログから実際の流れを再構成 | [As-Is / To-Be ビュー](/views/view-8-asis-tobe/)の Validation 面の発想 |
| Coordination Theory | 分析理論 | 活動間の依存関係と調整メカニズムの学術的分析 | [依存関係ビュー](/views/view-4-dependency/)の4つの型、[動力学](/dynamics/theoretical-foundations/)の理論的基盤 |
| Activity Theory / STS | 分析理論 | 主体・道具・ルール・分業の関係で人間活動を捉える理論 | [Actor の定義](/execution/actor-and-responsibility/)と[AIの4ファセット](/execution/ai-four-facets/)の理論的根拠 |

<Aside type="caution">
APQC PCF は「流れを捉える」フレームワークではなく「仕事を分類する」体系。Coordination Theory / Activity Theory は学術的な分析理論であり、Scrum や DevOps のような実務テンプレートとは性質が異なる。これらを同列に「フレームワーク」と呼ぶと議論が噛み合わなくなる。
</Aside>

### 汎用枠組みの組み合わせ

実務ではこれらを単体で使うことは少ない。「SIPOC で全体像を掴み → フローチャートで手順を描き → VSM で待ちを特定し → PDCA で改善を回す」といった具合に、目的に応じて組み合わせるのが当たり前である。

しかし、これらの枠組みはいずれもソフトウェア開発固有の課題には踏み込まない。ライフサイクルの定義、テスト戦略、デプロイの自動化といった関心事には、エンジニアリング特化の枠組みが必要になる。

## エンジニアリング特化の枠組み

ソフトウェア開発の「どう進めるか」に踏み込むもの。汎用枠組みとは関心事が異なり、開発ライフサイクル・チーム運営・デリバリー・測定・組織設計といった領域をカバーする。

### ライフサイクル定義: SDLC / ISO 12207 / V-model

SDLC は開発フェーズの共通語彙。ISO/IEC/IEEE 12207 は30プロセスを定義する国際標準で、テーラリング（プロジェクトに応じた適用）を前提とする。V-model は要求と検証の対応関係を明示する。

| 強み | 限界 |
|---|---|
| 開発の全体像を関係者で揃えられる | 実行主体の設計（人 vs AI）は扱わない |
| ISO 12207 はテーラリング可能 | 裁量レベルの概念はない |
| EU AI Act 等の規制が参照を加速 | AI前提の粒度分解は不足 |

<Aside>
ISO/IEC/IEEE 12207 は本モデルに最も近い既存標準である。テーラリング可能な参照モデルという思想が共通する。両者の差分は「実行主体の設計」「裁量レベル」「多視点ビュー」にある。
</Aside>

### チーム運営: Scrum / Kanban

Scrum は時間制約（スプリント）付きの反復的フレームワーク。Kanban はWIP制限と連続フローによるフロー管理手法。

| 強み | 限界 |
|---|---|
| チームの仕事の進め方を具体的に定義 | ライフサイクル全体は扱わない |
| Kanban のフロー指標は測定に有用 | AIエージェントの参加は想定していない |

### デリバリーモデル: DevOps / CI/CD

開発→運用の連続的なデリバリーフロー。2025年時点で事実上の業界標準。

| 強み | 限界 |
|---|---|
| 自動化駆動で高速なフィードバック | 上流（発見・戦略・要件定義）が弱い |
| 運用を含めた全体最適化 | 実行主体の設計は扱わない |

### ゲートモデル: Stage-Gate

Robert Cooper 提唱の新製品開発モデル。ステージ（活動）+ ゲート（Go/Kill 意思決定）で構成。

| 強み | 限界 |
|---|---|
| リスク管理とリソース配分が明確 | ゲート判定は人間の意思決定を前提 |
| 製造業で広く実績 | AIによる自動ゲート判定は想定外 |

### 測定: DORA / SPACE / DevEx

DORA Metrics はデリバリーの速度と安定性を5指標で計測するデファクトスタンダード。SPACE は開発者生産性の多次元評価（Satisfaction, Performance, Activity, Communication, Efficiency）。DevEx は開発者がフロー状態に入れる環境の設計に焦点を当てる。

| 強み | 限界 |
|---|---|
| 「速度と安定性はトレードオフではない」を実証 | AIエージェントの貢献をどう計測するかは未定義 |
| 多次元的で歪みにくい | ライフサイクル設計や責任設計は扱わない |

### 組織設計: Team Topologies

Matthew Skelton & Manuel Pais 提唱。4つのチームタイプ（Stream-aligned, Platform, Enabling, Complicated Subsystem）と認知負荷の管理を中心に、チーム構造と価値の流れの関係を設計する。

| 強み | 限界 |
|---|---|
| 認知負荷の管理が核心概念 | AIエージェントがどのチームタイプに入るかは未定義 |
| コンウェイの法則の逆適用 | ライフサイクルの段階設計は扱わない |

### 価値接続: Flow Framework

Mik Kersten 提唱。4つのフローアイテム（Features, Defects, Risks, Debt）でエンジニアリング指標とビジネス成果を接続する。

| 強み | 限界 |
|---|---|
| プロジェクト→プロダクトのシフトを体系化 | 実行レベルの設計は扱わない |
| リソース配分のトレードオフを可視化 | チーム・個人レベルのフローは対象外 |

### エンジニアリング枠組みの組み合わせ

汎用枠組みと同様に、エンジニアリング特化の枠組みも組み合わせて使うのが前提である。「SDLC でライフサイクルを定義し → Scrum でスプリントを回し → DevOps / CI/CD でデリバリーを自動化し → DORA で健全性を測定する」というのが典型的な組み合わせである。

さらに汎用枠組みとも組み合わさる。たとえば「VSM で開発フロー全体のボトルネックを特定し → DevOps で自動化を進め → PDCA で改善を回す」といった使い方は珍しくない。

## AI時代に加わる観点

人やシステムに加えて、AIエージェントという少し異なる性質を持つ実行主体が現れた。自律的に判断し、コードを書き、タスクを完了する——しかし責任は取れない。この新しい実行主体をワークフローに組み込むために、以下の観点を加える必要がある。

| 観点 | なぜ必要か |
|---|---|
| **時間軸の構造化** | AIが入り込む粒度まで仕事の段階を分解する必要がある |
| **実行主体と責任主体の分離** | AIが実行しても、責任は人に残る。この構造を設計する必要がある |
| **裁量レベルの設計** | 「全部任せる/任せない」ではなく、段階的に委譲度を設計する必要がある |
| **複数の観点からの統合的な可視化** | 流れ・依存・制御・測定を統合的に捉える必要がある |

### マッピング: 既存枠組み × 4つの観点

各枠組みが上記4つの観点にどこまで対応するかを整理する。

| 枠組み | 時間軸の構造化 | 実行/責任主体の分離 | 裁量レベルの設計 | 複数観点の統合 |
|---|---|---|---|---|
| PDCA / OODA | × | × | × | × |
| BPMN | △ フロー記述は可能 | × | × | × |
| Value Stream Mapping | △ 待ちと流れは見える | × | × | × |
| Service Blueprint | × | × | × | △ 表/裏の視点 |
| Coordination Theory | × | × | × | △ 依存関係の視点 |
| Activity Theory / STS | × | △ 主体と道具の分離 | × | △ 多面的分析 |
| **ISO 12207** | **◎ 30プロセス定義** | △ 役割は定義する | × | × |
| **SDLC / V-model** | **○ フェーズ定義** | × | × | × |
| **Scrum / Kanban** | △ スプリント/フロー | △ ロールは定義する | × | × |
| **DevOps / CI/CD** | ○ パイプライン | × | × | × |
| **Stage-Gate** | ○ ステージ+ゲート | △ ゲート判定者 | × | × |
| **DORA / SPACE** | × | × | × | △ 多次元指標 |
| **Team Topologies** | × | △ チーム責務 | × | × |
| **Flow Framework** | × | × | × | △ 価値視点 |

**凡例:** ◎ 主目的として扱う / ○ 部分的に扱う / △ 関連する視点はあるが不十分 / × 扱わない

この表が示すのは、個々の枠組みはそれぞれの目的に対して機能するが、AI時代に必要な4つの観点を揃えた枠組みはまだ存在しないということである。

## 既存の知見を、AIエージェントを前提とした枠組みに組み直す

本モデルは、既存の枠組みの知見にAI時代の観点を加え、**ワークフロー設計の枠組みとして組み直した参照モデル**である。既存の種別のどれか一つに収まるものではなく、複数の種別を組み合わせている。

| 構成要素 | 組み込んだ既存の知見 | 新たに加えた観点 |
|---|---|---|
| ライフサイクル + L2分解 | SDLC, ISO 12207 | AI前提の粒度分解 |
| ステップ構成 | RACI, DORA | 実行主体/責任主体の分離、制御環境・測定を各段階に重ねる |
| ビュー | VSM, Team Topologies | 共通の基底モデルから目的別に投影 |
| テーラリング | ISO 12207 | 語彙と構造だけを固定し、中身はコンテキストに応じて埋める |

## この整理から得られた教訓

既存フレームワークの調査から、以下の教訓が得られた。これらは本モデルの設計判断の根拠になっている。

| 教訓 | 本モデルへの反映 |
|---|---|
| 単一のフレームワークでは足りない | [複数ビュー](/views/why-multiple-views/)で投影する構造を採用 |
| ライフサイクルだけでは不十分 | ライフサイクルを主軸にしつつ、ステップ構成を重ねる構造を採用 |
| 要素だけでなく関係と動力学が必要 | [依存関係ビュー](/views/view-4-dependency/)と[動力学セクション](/dynamics/why-dynamics/)を追加 |
| 理想図だけでは不十分 | [As-Is / To-Be / Validation](/views/view-8-asis-tobe/) の3面構成を採用 |

<Aside>
これらの教訓の詳細な検討過程は[概念の導出過程](/appendix/derivation-process/)で解説している。
</Aside>
