## なぜ基底モデルが必要か

ソフトウェアエンジニアリングの仕事を語るとき、「誰が」「何を」「どの順序で」「何を使って」といった要素が暗黙のまま混在しがちである。ライフサイクルは時間軸、主体・役割は責任や分担、制御・基盤は仕事を支える条件であり、これらを同じ「軸」として並べるには無理がある。

基底モデルは、仕事を記述するための**最小限の共通語彙**を定義する。この語彙があることで、どのライフサイクルステップについても同じ構造で記述・比較・設計ができるようになる。

## 要素

仕事を記述する最小単位として、以下の要素を置く。

| # | 要素 | 英名 | 問い |
|---|---|---|---|
| 1 | 対象 | Object | 何を変えたいのか？ |
| 2 | ライフサイクルステップ | Lifecycle Step | 時間軸上のどの段階か？ |
| 3 | 主体 | Actor | 誰が仕事を前進させるか？ |
| 4 | 媒介物 | Artifact | 何を介して仕事が流れるか？ |
| 5 | 依存 | Dependency | 何が何に依存しているか？ |
| 6 | 制約 | Control | 安全性・再現性・監査性をどう担保するか？ |
| 7 | 指標 | Metric | 仕事の流れや品質をどう評価するか？ |

<Aside type="tip">
要素は「何を定義すべきか」の枠であり、具体的な値は規定しない。値はコンテキストに応じて埋める調整項目である。
</Aside>

## 各要素の説明

### 1. Object（対象）

仕事の目的物。課題、目的、価値、対象を指す。「何のためにこの仕事が存在するのか」を明確にする起点である。

たとえば Implementation ステップにおける Object は「仕様で定義された振る舞いをコードとして実現すること」であり、Verification & Review における Object は「変更が品質基準を満たしているかを確認すること」である。

### 2. Lifecycle Step（ライフサイクルステップ）

時間軸上の段階。本モデルでは L1 としてライフサイクルを定義し、必要に応じて L2 に分解する。各ステップは開始条件と終了条件を持ち、前のステップの出力が次のステップの入力になる。

### 3. Actor（主体）

仕事の状態を前進させる行為単位。以下の4種を含める。

| 種類 | 説明 |
|---|---|
| Human | 人間の開発者、レビュアー、PO など |
| AI Agent | コードを書く、レビューする、テストを生成するなどの実行を担う AI |
| Automation / System | CI/CD、リンター、自動テストなどの自動化された仕組み |
| External Actor | 外部ベンダー、顧客、規制機関など |

<Aside type="caution">
Tool、Skill、Prompt、Document、Knowledge base、Harness は Actor に含めない。これらは主体ではなく、能力・媒介物・制御環境として扱う。
</Aside>

#### AI の位置づけ

AI は主に**実行主体**として入る。ただし、**責任主体ではない**。

- 実行主体: AI Agent になりうる
- 責任主体: 通常は Human または Team に残る
- AI ネイティブ化とは: 責任移譲ではなく、**裁量移譲の範囲設計**

### 4. Artifact（媒介物）

仕事を媒介するもの。あるステップの出力が、次のステップの入力として機能する。要件定義書、設計書、ソースコード、テスト結果、デプロイパッケージなどが該当する。

### 5. Dependency（依存）

何が何に依存しているかの関係。ステップ間の前後関係、ツール間の連携、チーム間の待ち合わせなどが含まれる。依存の可視化は、ボトルネックの発見と解消に直結する。

### 6. Control（制約）

安全性・再現性・監査性を支える制約。ゲート条件、自動検証、承認フロー、ロールバック手順などが該当する。AI の裁量を広げるほど、Control の設計が重要になる。

### 7. Metric（指標）

仕事の流れや品質を評価する指標。リードタイム、変更失敗率、レビュー待ち時間、テストカバレッジなどが含まれる。現状の計測（記述）にも、目標の設定（設計）にも使える。

## 具体例: Implementation を7要素で記述する

L1「Implementation」の L2「変更実装」を7要素で記述すると、以下のようになる。

| 要素 | 記述例 |
|---|---|
| **Object** | 設計で決定された方式に基づき、変更対象のコードを実装する |
| **Lifecycle Step** | Implementation > 変更実装 |
| **Actor** | 実行主体: AI Agent（Claude Code）、責任主体: Human（開発者） |
| **Artifact** | 入力: 設計書、タスク分解表。出力: 変更済みソースコード |
| **Dependency** | Design ステップの完了、文脈収集（L2）の完了 |
| **Control** | リンター自動実行、型チェック、差分の範囲制限（変更対象ファイルのみ） |
| **Metric** | 実装所要時間、初回パス率（自動検証を何回で通過するか） |

<Aside>
この例は一つのインスタンス化の結果である。Actor に誰を割り当てるか、Control にどの検証を入れるかは、チームやプロジェクトのコンテキストによって変わる。
</Aside>

## なぜ複数ビューが必要か

7要素を1枚の図に押し込むと、情報量が多すぎて理解が困難になる。仕事を理解するには、少なくとも以下のように異なる観点で切り分ける必要がある。

- 時間の流れ（ライフサイクルビュー）
- 誰がどう進めるか（実行設計ビュー）
- 何を介して流れるか（アーティファクトビュー）
- 何に依存しているか（依存関係ビュー）
- 何が裏側で支えているか（制御環境ビュー）
- どう測るか（測定ビュー）
- 現実と理想がどれだけズレているか（As-Is / To-Be ビュー）

本モデルでは、これらを**ビュー**として定義している。各ビューは基底要素のうち特定の側面にフォーカスし、他のビューと組み合わせて全体像を構成する。
